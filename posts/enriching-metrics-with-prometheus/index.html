Enriching Metrics with Prometheus

June 27, 2023

    Metrics without context can be hard to explore and even harder to understand, especially if you’re in the depths of troubleshooting a difficult issue. Prometheus provides us with a useful pattern that allows you to match metadata with scraped metrics and attach new labels.

    Why?

    Useful details can be added to metrics by pulling from a database, source of truth platform such as Netbox or other inventory systems. Say you’re scraping node-exporters that are exposing data from a data centre of hypervisors, it might be useful to add additional metadata about each server’s geography or data centre location. This would allow for more useful aggregate queries, such as:

    avg((node_memory_MemAvailable_bytes_e{region="Aus/Sydney"} / node_memory_MemTotal_bytes_e{region="Aus/Sydney"} * 100))

    This query will show us the average memory utilisation across all servers in the Aus/Sydney geographic region. A query like this can be useful for long-term capacity planning and fleet optimisation.

    How

    There are two pieces to enriching Prometheus metrics, firstly pulling in the metadata in a useable form. In the case of the above, you’ll need to build or set up an exporter that produces the following metrics:

    hardware_inventory{instance="hostname01", region="Aus/Sydney", pod="2"} 1
hardware_inventory{instance="hostname02", region="Aus/Sydney", pod="2"} 1
hardware_inventory{instance="hostname03", region="Aus/Sydney", pod="2"} 1
hardware_inventory{instance="hostname04", region="Aus/Sydney", pod="2"} 1

    The metric value of 1 is important here, it’ll make this enrichment work later. Include any other labels against the hostname you think could be useful for aggregate queries.

    You may need to write a small application to query your metadata source and expose this information in the above format. In the case of Netbox, you can create a custom output format and avoid needing to write anything custom.

    The second piece is a Prometheus Recording Rule to join the metadata with our node-exporter metrics and output these to a new metric.

    groups:
  - name: hardware_enrichment
    interval: 30s
    rules:
      - record: node_memory_MemAvailable_bytes_e
        expr: node_memory_MemAvailable_bytes * on (instance) group_left (region, pod) hardware_inventory

    This rule will result in the addition of the region and pod labels in a metric like this:

    node_memory_MemAvailable_bytes_e{instance="hostname01", region="Aus/Sydney", pod="2"} 2218521241
node_memory_MemAvailable_bytes_e{instance="hostname02", region="Aus/Sydney", pod="2"} 2019368704
node_memory_MemAvailable_bytes_e{instance="hostname03", region="Aus/Sydney", pod="2"} 2423835744
node_memory_MemAvailable_bytes_e{instance="hostname04", region="Aus/Sydney", pod="2"} 2032872346

    Now we can query at the aggregate memory usage across the fleet, either filtering by a label or grouping to show the average per region.

    avg(node_memory_MemAvailable_bytes_e{region="Aus/Sydney"})
avg(node_memory_MemAvailable_bytes_e) by (region)
